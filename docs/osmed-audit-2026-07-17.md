## Исполнительное резюме

OSMed выглядит как крупный desktop-продукт на Electron + React + Vite с большим количеством модулей: CRM, пациенты, платежи, чат, мессенджеры, парсеры, VPN, ИИ-офис, офисные документы, удаленный доступ и beta-инструменты. Основная проблема сейчас в том, что часть модулей выглядит как готовый продукт, но фактически работает на локальном JSON/seed-данных, переключателях статусов или внешних ссылках. Есть серьезные риски безопасности: простые пароли в seed, хранение токенов без шифрования, широкие разрешения Electron WebView и небезопасный `openExternal`.

## Критичные проблемы

### 1. Слабая модель авторизации

Файлы:

- `/Users/polzovatel/программы/Азимут OSMed/src/components/LoginGate.tsx:18`
- `/Users/polzovatel/программы/Азимут OSMed/src/data/seed.ts:186`

Проблема:

- логин сверяет пароль как обычную строку;
- seed содержит дефолтные пароли `director/osmed2026`, `admin/admin2026`, `hr/hr2026`;
- нет хэширования, соли, rate limit, блокировки попыток, смены первичного пароля;
- сессия хранится в общем состоянии данных.

Что сделать:

- заменить пароль на `passwordHash` через Argon2id/bcrypt/scrypt;
- при первом запуске требовать смену дефолтных паролей;
- хранить session/access state отдельно от бизнес-данных;
- добавить rate limit и аудит входов;
- добавить роли и права на уровне действий, не только на уровне разделов.

### 2. Electron разрешает все permissions во встроенных сессиях

Файл:

- `/Users/polzovatel/программы/Азимут OSMed/electron/main.js:63`

Проблема:

`ses.setPermissionRequestHandler((_wc, _permission, callback) => callback(true));`

Это автоматически разрешает запросы разрешений во встроенных сессиях. Для приложения с мессенджерами, WebView, внешними сайтами и медицинскими данными это риск камеры, микрофона, геолокации, уведомлений и других browser-permissions.

Что сделать:

- сделать allowlist permissions;
- разрешать камеру/микрофон только для конкретных trusted origins;
- по умолчанию `callback(false)`;
- логировать запросы;
- отдельно обрабатывать Telegram/WhatsApp/VK, а не все домены.

### 3. `openExternal` открывает любые URL без проверки

Файл:

- `/Users/polzovatel/программы/Азимут OSMed/electron/main.js:572`

Проблема:

Renderer может передать любой URL в `shell.openExternal(url)`. Без проверки схемы возможны опасные `file:`, `javascript:`, нестандартные URL-схемы или phishing-flow.

Что сделать:

- разрешить только `https:` и при необходимости `mailto:`/`tel:`;
- блокировать `file:`, `javascript:`, `data:`, custom schemes;
- для внешних доменов показывать подтверждение;
- централизовать безопасный helper `safeOpenExternal`.

### 4. Токены и сессии мессенджеров хранятся открытым текстом

Файлы:

- `/Users/polzovatel/программы/Азимут OSMed/electron/messengers/telegram.js:87`
- `/Users/polzovatel/программы/Азимут OSMed/electron/messengers/vk.js:18`

Проблема:

Telegram config/session и VK token пишутся в файлы под `userData` без шифрования. На рабочем ПК клиники это превращает локальный профиль пользователя в хранилище доступа к мессенджерам.

Что сделать:

- использовать `safeStorage` Electron или OS keychain;
- разделить секреты и обычные настройки;
- добавить кнопку “сбросить сессию”;
- не писать токены в общий JSON;
- ограничить права файлов.

### 5. Потеря данных из-за async save без await

Файл:

- `/Users/polzovatel/программы/Азимут OSMed/src/lib/store.ts:59`

Проблема:

`updateStore` мутирует объект и вызывает `void persist(data)`, не ожидая сохранения и не показывая ошибку. Если приложение закрыть сразу после действия или запись упадет, пользователь думает, что данные сохранены, но это не гарантировано.

Что сделать:

- вернуть `Promise` из mutation-операций;
- показывать состояние “сохранено/ошибка”;
- ставить очередь сохранений;
- добавить atomic write: временный файл -> rename;
- добавить журнал изменений/backup.

## Высокие проблемы

### 6. WebView/WebContents без жесткой политики доменов

Файлы:

- `/Users/polzovatel/программы/Азимут OSMed/electron/main.js:481`
- `/Users/polzovatel/программы/Азимут OSMed/src/components/WebViewFrame.tsx:50`

Проблема:

Приложение встраивает внешние сайты и мессенджеры. Даже при `nodeIntegration: false` остаются риски phishing, popups, downloads, permissions, navigation и mixed-content поведения.

Что сделать:

- ввести allowlist доменов по разделам;
- блокировать неожиданные редиректы;
- обрабатывать downloads;
- отключить `allowpopups`, где он не нужен;
- добавить явную политику `setWindowOpenHandler`.

### 7. VPN WireGuard сейчас является stub-режимом

Файл:

- `/Users/polzovatel/программы/Азимут OSMed/electron/main.js:392`

Проблема:

Если конфиг содержит `[Interface]`, приложение сразу ставит `connected: true` и возвращает `mode: wireguard-stub`, не поднимая реальное VPN-подключение. Для пользователя это выглядит как работающий VPN, но трафик может никуда не идти.

Что сделать:

- честно показывать “WireGuard не реализован”;
- либо встроить `wireguard-go`/system WireGuard;
- проверять реальный интерфейс/маршрут/public IP после подключения.

### 8. OpenVPN подключение может быть ложноположительным

Файл:

- `/Users/polzovatel/программы/Азимут OSMed/electron/main.js:426`

Проблема:

Если OpenVPN за 2.8 секунды не написал `Initialization Sequence Completed`, код все равно вызывает `finish(true)`. Это может пометить неуспешное подключение как успешное.

Что сделать:

- считать успешным только явный success event;
- timeout должен возвращать ошибку/ожидание;
- после подключения проверять public IP и маршрут.

### 9. Парсеры поисковиков хрупкие и могут ломаться

Файл:

- `/Users/polzovatel/программы/Азимут OSMed/electron/parsers/runner.js:88`
- `/Users/polzovatel/программы/Азимут OSMed/electron/parsers/runner.js:112`

Проблема:

Без API-ключей Google/Yandex fallback-парсинг идет по HTML выдачи через регулярные выражения. Это нестабильно, часто ловит капчу и может нарушать правила источников.

Что сделать:

- для Google использовать SerpAPI/официальные провайдеры;
- для Yandex использовать XML API;
- добавить backoff, rate limit, логирование и понятные ошибки;
- хранить результаты с датой и источником.

### 10. “Офис Lite” хранит HTML и вставляет его через `dangerouslySetInnerHTML`

Файл:

- `/Users/polzovatel/программы/Азимут OSMed/src/pages/OfficeLite.tsx:47`
- `/Users/polzovatel/программы/Азимут OSMed/src/pages/OfficeLite.tsx:206`

Проблема:

Документ сохраняется как `innerHTML`, затем вставляется обратно через `dangerouslySetInnerHTML`. Если HTML попадет из внешнего источника или через импорт, можно получить XSS внутри приложения.

Что сделать:

- санитизировать HTML через DOMPurify;
- ограничить набор тегов/атрибутов;
- для документов хранить структурированную модель или Markdown;
- экспорт DOCX/PDF делать серверно/через библиотеку, а не через HTML-печать.
